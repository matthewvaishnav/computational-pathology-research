# Requirements Document: WSI Processing Pipeline

## Introduction

This document specifies requirements for implementing a complete Whole Slide Image (WSI) processing pipeline for HistoCore. The pipeline will enable processing of real hospital slides in clinical formats (.svs, .tiff, .ndpi) by integrating OpenSlide for WSI reading, implementing patch extraction at multiple magnification levels, handling gigapixel images efficiently, supporting tissue detection and background filtering, generating feature embeddings for MIL models, caching processed features for reuse, and supporting batch processing of multiple slides.

The current HistoCore implementation uses a feature-cache baseline with pre-extracted HDF5 features. This WSI processing pipeline is a critical blocker for clinical deployment, as it will enable HistoCore to process real hospital slides directly from PACS systems.

## Glossary

- **WSI_Reader**: Component that reads whole-slide images using OpenSlide library
- **Patch_Extractor**: Component that extracts patches/tiles from WSI at specified coordinates and magnification levels
- **Tissue_Detector**: Component that identifies tissue regions and filters background areas
- **Feature_Generator**: Component that generates feature embeddings from patches using pretrained encoders
- **Feature_Cache**: Component that stores and retrieves processed features in HDF5 format
- **Batch_Processor**: Component that processes multiple slides in parallel or sequence
- **Pyramid_Level**: Magnification level in multi-resolution WSI pyramid (level 0 = highest resolution)
- **Patch**: Rectangular image region extracted from WSI at specific coordinates and size
- **Tissue_Mask**: Binary mask indicating tissue vs background regions
- **Feature_Embedding**: Vector representation of patch generated by neural network encoder
- **HDF5_Cache**: Hierarchical Data Format file storing features, coordinates, and metadata
- **PACS**: Picture Archiving and Communication System used in hospitals
- **Magnification**: Objective lens power (e.g., 20x, 40x) determining image resolution
- **MPP**: Microns per pixel, physical resolution of WSI
- **DICOM_WSI**: Digital Imaging and Communications in Medicine format for whole-slide images

## Requirements

### Requirement 1: WSI File Format Support

**User Story:** As a computational pathologist, I want to read WSI files in clinical formats, so that I can process slides from hospital PACS systems.

#### Acceptance Criteria

1. WHEN a .svs file (Aperio format) is provided, THE WSI_Reader SHALL open the file and extract metadata
2. WHEN a .tiff file (generic TIFF format) is provided, THE WSI_Reader SHALL open the file and extract metadata
3. WHEN a .ndpi file (Hamamatsu format) is provided, THE WSI_Reader SHALL open the file and extract metadata
4. WHEN a DICOM WSI file is provided, THE WSI_Reader SHALL open the file and extract metadata
5. WHEN an unsupported file format is provided, THE WSI_Reader SHALL return an error message indicating the unsupported format
6. THE WSI_Reader SHALL extract slide dimensions at level 0 (width and height in pixels)
7. THE WSI_Reader SHALL extract the number of pyramid levels available
8. THE WSI_Reader SHALL extract magnification information from slide metadata
9. THE WSI_Reader SHALL extract microns per pixel (MPP) from slide metadata
10. WHEN metadata fields are missing, THE WSI_Reader SHALL return default values and log a warning

### Requirement 2: Multi-Resolution Pyramid Support

**User Story:** As a researcher, I want to read patches at different magnification levels, so that I can analyze tissue at appropriate resolutions.

#### Acceptance Criteria

1. THE WSI_Reader SHALL provide access to all pyramid levels in the WSI file
2. THE WSI_Reader SHALL return downsample factors for each pyramid level
3. THE WSI_Reader SHALL return dimensions for each pyramid level
4. WHEN a patch is requested at a specific pyramid level, THE Patch_Extractor SHALL read from that level
5. WHEN coordinates are provided at level 0, THE Patch_Extractor SHALL convert them to the requested pyramid level
6. THE Patch_Extractor SHALL maintain spatial correspondence between patches at different pyramid levels
7. WHEN a pyramid level does not exist, THE WSI_Reader SHALL return an error indicating the invalid level

### Requirement 3: Patch Extraction

**User Story:** As a machine learning engineer, I want to extract patches from WSI at configurable sizes, so that I can feed them to neural networks.

#### Acceptance Criteria

1. WHEN coordinates and patch size are provided, THE Patch_Extractor SHALL extract a patch at those coordinates
2. THE Patch_Extractor SHALL support patch sizes of 256x256 pixels
3. THE Patch_Extractor SHALL support patch sizes of 512x512 pixels
4. THE Patch_Extractor SHALL support configurable patch sizes between 64 and 2048 pixels
5. THE Patch_Extractor SHALL return patches in RGB format (3 channels)
6. WHEN coordinates are outside slide boundaries, THE Patch_Extractor SHALL return an error indicating out-of-bounds coordinates
7. THE Patch_Extractor SHALL extract patches with configurable stride for overlapping or non-overlapping sampling
8. THE Patch_Extractor SHALL record the (x, y) coordinates for each extracted patch
9. THE Patch_Extractor SHALL record the pyramid level for each extracted patch

### Requirement 4: Tissue Detection and Background Filtering

**User Story:** As a computational pathologist, I want to filter out background regions, so that I only process tissue-containing patches.

#### Acceptance Criteria

1. WHEN a patch is extracted, THE Tissue_Detector SHALL determine if the patch contains tissue
2. THE Tissue_Detector SHALL use Otsu thresholding for tissue detection
3. THE Tissue_Detector SHALL calculate the percentage of tissue pixels in each patch
4. WHEN tissue percentage is below a configurable threshold, THE Tissue_Detector SHALL mark the patch as background
5. WHEN tissue percentage is above the threshold, THE Tissue_Detector SHALL mark the patch as tissue
6. THE Tissue_Detector SHALL support configurable tissue percentage thresholds between 0.0 and 1.0
7. THE Tissue_Detector SHALL generate a binary tissue mask at a lower pyramid level for efficiency
8. WHERE deep learning tissue detection is enabled, THE Tissue_Detector SHALL use a pretrained tissue segmentation model
9. THE Tissue_Detector SHALL filter patches before feature extraction to reduce computation

### Requirement 5: Memory-Efficient Streaming

**User Story:** As a developer, I want to process gigapixel images without loading the entire slide into memory, so that I can handle large WSI files on limited hardware.

#### Acceptance Criteria

1. THE WSI_Reader SHALL read patches on-demand without loading the entire slide into memory
2. THE Patch_Extractor SHALL process patches in streaming fashion without accumulating all patches in memory
3. WHEN processing a slide with 100,000 x 100,000 pixels, THE WSI_Reader SHALL use less than 4GB of RAM
4. THE Batch_Processor SHALL process patches in batches to limit memory usage
5. THE Batch_Processor SHALL support configurable batch sizes between 1 and 1024 patches
6. WHEN memory usage exceeds a configurable threshold, THE Batch_Processor SHALL reduce batch size automatically
7. THE WSI_Reader SHALL close file handles after processing to release resources

### Requirement 6: Feature Embedding Generation

**User Story:** As a machine learning engineer, I want to generate feature embeddings from patches, so that I can train MIL models.

#### Acceptance Criteria

1. WHEN a patch is provided, THE Feature_Generator SHALL generate a feature embedding vector
2. THE Feature_Generator SHALL support ResNet-50 as a pretrained encoder
3. THE Feature_Generator SHALL support DenseNet-121 as a pretrained encoder
4. THE Feature_Generator SHALL support EfficientNet-B0 as a pretrained encoder
5. THE Feature_Generator SHALL support configurable pretrained encoders from torchvision and timm
6. THE Feature_Generator SHALL extract features from the penultimate layer of the encoder
7. THE Feature_Generator SHALL return feature embeddings as float32 tensors
8. THE Feature_Generator SHALL process patches in batches for GPU efficiency
9. WHERE GPU is available, THE Feature_Generator SHALL use GPU acceleration
10. WHEN GPU is not available, THE Feature_Generator SHALL use CPU processing

### Requirement 7: Feature Caching

**User Story:** As a researcher, I want to cache processed features, so that I can reuse them without reprocessing slides.

#### Acceptance Criteria

1. WHEN features are generated for a slide, THE Feature_Cache SHALL store them in HDF5 format
2. THE Feature_Cache SHALL store feature embeddings in a dataset named "features"
3. THE Feature_Cache SHALL store patch coordinates in a dataset named "coordinates"
4. THE Feature_Cache SHALL store slide metadata as HDF5 attributes
5. THE Feature_Cache SHALL store pyramid level information for each patch
6. WHEN a cached feature file exists, THE Feature_Cache SHALL load features from the file
7. WHEN a cached feature file is corrupted, THE Feature_Cache SHALL return an error and suggest regeneration
8. THE Feature_Cache SHALL validate HDF5 file structure before loading
9. THE Feature_Cache SHALL support compression to reduce storage size
10. THE Feature_Cache SHALL store feature extraction timestamp for cache invalidation

### Requirement 8: Batch Processing

**User Story:** As a researcher, I want to process multiple slides in batch, so that I can efficiently prepare datasets.

#### Acceptance Criteria

1. WHEN a list of slide paths is provided, THE Batch_Processor SHALL process all slides sequentially
2. THE Batch_Processor SHALL support parallel processing of multiple slides
3. THE Batch_Processor SHALL support configurable number of worker processes between 1 and 16
4. WHEN a slide fails to process, THE Batch_Processor SHALL log the error and continue with remaining slides
5. THE Batch_Processor SHALL report progress for each slide (percentage complete)
6. THE Batch_Processor SHALL report estimated time remaining for batch processing
7. THE Batch_Processor SHALL generate a summary report after batch processing completes
8. THE Batch_Processor SHALL track successfully processed slides and failed slides
9. WHERE GPU is available, THE Batch_Processor SHALL distribute slides across available GPUs

### Requirement 9: Performance Requirements

**User Story:** As a clinical user, I want fast processing times, so that I can integrate WSI processing into clinical workflows.

#### Acceptance Criteria

1. WHEN processing a 40x magnification slide, THE Batch_Processor SHALL complete within 10 minutes
2. WHEN processing a slide with 100,000 x 100,000 pixels, THE Batch_Processor SHALL complete within 15 minutes
3. THE Patch_Extractor SHALL extract patches at a rate of at least 100 patches per second
4. WHERE GPU is available, THE Feature_Generator SHALL process at least 500 patches per second
5. WHEN GPU is not available, THE Feature_Generator SHALL process at least 50 patches per second
6. THE Tissue_Detector SHALL process tissue masks at a rate of at least 1000 patches per second
7. THE Feature_Cache SHALL write HDF5 files at a rate of at least 10 MB per second

### Requirement 10: Quality Control

**User Story:** As a computational pathologist, I want quality control checks, so that I can identify problematic slides.

#### Acceptance Criteria

1. WHEN a slide is processed, THE Batch_Processor SHALL detect blurry patches
2. THE Batch_Processor SHALL calculate blur scores using Laplacian variance
3. WHEN blur score is below a configurable threshold, THE Batch_Processor SHALL mark the patch as blurry
4. THE Batch_Processor SHALL detect artifacts (pen marks, bubbles, folds)
5. THE Batch_Processor SHALL calculate tissue coverage percentage for the entire slide
6. WHEN tissue coverage is below 10 percent, THE Batch_Processor SHALL log a warning
7. THE Batch_Processor SHALL validate that extracted patches have correct dimensions
8. THE Batch_Processor SHALL validate that feature embeddings have correct dimensions
9. THE Batch_Processor SHALL generate a quality control report for each slide

### Requirement 11: Clinical Metadata Preservation

**User Story:** As a clinical researcher, I want to preserve patient metadata, so that I can maintain data provenance.

#### Acceptance Criteria

1. WHEN a slide is processed, THE WSI_Reader SHALL extract patient ID from slide metadata
2. THE WSI_Reader SHALL extract scan date from slide metadata
3. THE WSI_Reader SHALL extract scanner model from slide metadata
4. THE Feature_Cache SHALL store patient metadata in HDF5 attributes
5. THE Feature_Cache SHALL store scan parameters (magnification, MPP) in HDF5 attributes
6. THE Feature_Cache SHALL store processing parameters (patch size, stride) in HDF5 attributes
7. THE Feature_Cache SHALL store feature extractor model name and version in HDF5 attributes
8. WHEN patient metadata contains PHI, THE Feature_Cache SHALL support anonymization
9. THE Feature_Cache SHALL maintain metadata consistency across cached files

### Requirement 12: Error Handling and Robustness

**User Story:** As a developer, I want robust error handling, so that the pipeline can handle edge cases gracefully.

#### Acceptance Criteria

1. WHEN a WSI file is corrupted, THE WSI_Reader SHALL return a descriptive error message
2. WHEN a WSI file is missing, THE WSI_Reader SHALL return a file-not-found error
3. WHEN OpenSlide fails to open a file, THE WSI_Reader SHALL return the OpenSlide error message
4. WHEN disk space is insufficient, THE Feature_Cache SHALL return a disk-space error
5. WHEN GPU memory is exhausted, THE Feature_Generator SHALL fall back to CPU processing
6. WHEN a patch extraction fails, THE Patch_Extractor SHALL log the error and continue with remaining patches
7. THE Batch_Processor SHALL implement retry logic for transient failures
8. THE Batch_Processor SHALL support configurable maximum retry attempts between 0 and 5
9. WHEN all retries are exhausted, THE Batch_Processor SHALL mark the slide as failed and continue

### Requirement 13: Configuration and Extensibility

**User Story:** As a researcher, I want configurable pipeline parameters, so that I can adapt the pipeline to different use cases.

#### Acceptance Criteria

1. THE Batch_Processor SHALL support configuration via YAML files
2. THE Batch_Processor SHALL support configuration via Python dictionaries
3. THE Batch_Processor SHALL validate configuration parameters before processing
4. WHEN invalid configuration is provided, THE Batch_Processor SHALL return a descriptive error message
5. THE Feature_Generator SHALL support custom pretrained encoders via configuration
6. THE Tissue_Detector SHALL support custom tissue detection models via configuration
7. THE Patch_Extractor SHALL support custom patch extraction strategies via configuration
8. THE Feature_Cache SHALL support custom HDF5 dataset names via configuration

### Requirement 14: Integration with Existing HistoCore Components

**User Story:** As a HistoCore user, I want seamless integration with existing components, so that I can use the WSI pipeline with current workflows.

#### Acceptance Criteria

1. THE Feature_Cache SHALL generate HDF5 files compatible with existing CAMELYONSlideDataset class
2. THE Feature_Cache SHALL generate HDF5 files compatible with existing CAMELYONPatchDataset class
3. THE Feature_Cache SHALL store features in the same format as existing feature files
4. THE Feature_Cache SHALL store coordinates in the same format as existing coordinate files
5. WHEN WSI processing is complete, THE Batch_Processor SHALL generate a slide index JSON file
6. THE Batch_Processor SHALL generate slide index files compatible with CAMELYONSlideIndex class
7. THE Batch_Processor SHALL support integration with existing training scripts (train_camelyon.py)
8. THE Batch_Processor SHALL support integration with existing evaluation scripts (evaluate_camelyon.py)

### Requirement 15: Documentation and Examples

**User Story:** As a new user, I want clear documentation and examples, so that I can quickly start using the WSI pipeline.

#### Acceptance Criteria

1. THE WSI_Reader SHALL include docstrings for all public methods
2. THE Patch_Extractor SHALL include docstrings for all public methods
3. THE Tissue_Detector SHALL include docstrings for all public methods
4. THE Feature_Generator SHALL include docstrings for all public methods
5. THE Feature_Cache SHALL include docstrings for all public methods
6. THE Batch_Processor SHALL include docstrings for all public methods
7. THE WSI pipeline SHALL include a README with usage examples
8. THE WSI pipeline SHALL include example scripts for common use cases
9. THE WSI pipeline SHALL include a tutorial notebook demonstrating end-to-end processing
10. THE WSI pipeline SHALL include API documentation generated from docstrings
